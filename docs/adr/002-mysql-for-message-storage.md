# ADR-002: 使用 MySQL 存储消息，不用时序数据库

## 背景

设计消息系统时，需要选择消息的持久化存储方案。消息数据有明显的时间属性（按时间排序、按时间范围查询），因此考虑过时序数据库（InfluxDB、TimescaleDB）。

## 决策

使用已有的 MySQL 存储消息，不引入时序数据库。

## 理由

**时序数据库不擅长 IM 的核心操作：**

| 操作 | 时序数据库 | MySQL |
|------|-----------|-------|
| 更新消息状态（已读/已送达） | ❌ 数据通常不可变 | ✅ UPDATE |
| 消息撤回（删除指定消息） | ❌ 不擅长 | ✅ DELETE |
| 按 userId 查历史消息 | ⚠️ 需要 tag 过滤 | ✅ 直接索引 |
| 消息+用户信息关联查询 | ❌ 无 JOIN | ✅ JOIN |

**额外理由：**
- 项目已有 MySQL，不引入新的基础设施，降低运维复杂度
- 当前规模（百万消息级别以下）MySQL 加索引完全够用
- 热数据走 Redis 缓存（最近 100 条），MySQL 只承压冷数据查询

**Discord 用 Cassandra 的场景（不适用于本项目）：**
超大群（数十万人）、每秒百万级写入、查询模式极其固定、几乎不需要更新。本项目早期不具备这个量级。

## 后续演进路径

| 阶段 | 方案 |
|------|------|
| 当前（百万消息以下） | MySQL + 索引 |
| 消息量增大 | 按月分表（messages_2024_01） |
| 千万级以上 | 迁移 TiDB（MySQL 协议兼容，代码几乎不改） |
| 超大规模（Discord 量级） | 考虑 Cassandra，届时重新评估 |
